    A tagged union of all possible typechecking errors produced by the typechecker.

    What it represents TypingError collects every error that can arise during:

    • type inference (inferType)
    • type checking (checkType)
    • kind checking (checkKind)
    • pattern matching (checkPattern, checkExhaustive)
    • trait resolution (checkTraitImplementation, checkTraitConstraints)
    • module importing (importModule)
    • type unification (unifyTypes)

    Each error variant provides precise information about what went wrong.

    Categories of errors

    Lookup / Missing definitions

    • UnboundTypeError — missing type or variable
    • MissingTraitImplError — missing trait instance
    • MissingMethodError — missing required trait method
    • MissingFieldTypeError — missing record field
    • MissingCaseTypeError — missing match case

    Kind errors

    • KindMismatchTypeError — kind mismatch
    • UnexpectedKindError — invalid or unexpected kind
    • NotATypeFunctionTypeError — type application on a non‑function kind

    Type errors

    • TypeMismatchTypeError — incompatible types
    • CyclicTypeError — infinite/cyclic type detected
    • WrongNumberOfDictsError — mismatched dictionary evidence arity

    Data shape errors

    • NotARecordTypeError — used as record but isn’t
    • NotATupleTypeError — used as tuple but isn’t
    • TupleIndexOutofBoundsTypeError — invalid tuple projection
    • NotAVariantTypeError — used as variant but isn’t
    • InvalidVariantTypeError — invalid variant label
    • ExtraCaseTypeError — extraneous match case not present in data

    Function errors

    • NotAFunctionTypeError — applying non‑function value

    Module/import errors

    • CircularImportError — dependency cycle during module import
    • DuplicateBindingError — conflicting name in context

    Why it’s useful Collecting all errors under a single type allows:

    • clean and uniform error handling through Result<TypingError, T>
    • pattern matching on error kinds
    • user‑friendly formatting via showError
    • IDE tooling and REPLs to surface precise diagnostics

    Where it is used

    • All high‑level APIs return Result<TypingError, T>
    • inferType, checkType, solveConstraints
    • importModule
    • showError understands every variant

    Example

    import {
      freshState, inferType, varTerm,
      showError
    } from "system-f-omega";

    const state = freshState();
    const res = inferType(state, varTerm("missing"));

    if ("err" in res) {
      console.log(showError(res.err));  // "Unbound identifier: missing"
    }
